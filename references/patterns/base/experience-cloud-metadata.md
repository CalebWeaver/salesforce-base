# Experience Cloud Metadata Deployment

Experience Cloud metadata is among the most complex in Salesforce. Deployment order matters, ExperienceBundles are large and conflict-prone, and guest user configuration requires coordinating profiles, sharing rules, and permission sets. Always retrieve before modifying.

## Metadata Types Overview

| Metadata Type | What It Controls | Deploy Order |
|---------------|-----------------|--------------|
| **Network** | Site configuration, member profiles, self-registration, guest user profile | 1st — must exist before ExperienceBundle |
| **CustomSite** | URL path, site type, site admin | 1st (alongside Network) |
| **ExperienceBundle** | Pages, routes, views, theme, branding, components | 2nd — requires Network |
| **NavigationMenu** | Site navigation menus | 2nd — references pages in ExperienceBundle |
| **NavigationMenuItem** | Individual menu links | 2nd — child of NavigationMenu |
| **Audience** | Audience targeting criteria | Any time |
| **Profile** (Guest User) | Object/field permissions for unauthenticated users | Any time, but must align with Network config |

## Retrieving Experience Cloud Metadata

Always retrieve by name — source tracking is unreliable for ExperienceBundles because auto-generated files create constant noise.

```bash
# Retrieve the full ExperienceBundle for a site named "Customer_Portal"
sf project retrieve start --metadata ExperienceBundle:Customer_Portal

# Retrieve Network configuration
sf project retrieve start --metadata Network:Customer_Portal

# Retrieve NavigationMenus
sf project retrieve start --metadata NavigationMenu:Customer_Portal_Navigation

# Retrieve guest user profile (naming convention: "{SiteName} Profile")
sf project retrieve start --metadata Profile:Customer_Portal\ Profile

# Retrieve everything at once
sf project retrieve start \
  --metadata ExperienceBundle:Customer_Portal \
  --metadata Network:Customer_Portal \
  --metadata NavigationMenu:Customer_Portal_Navigation \
  --metadata Profile:Customer_Portal\ Profile
```

## ExperienceBundle Directory Structure

After retrieval, the ExperienceBundle lands in `force-app/main/default/experiences/{SiteName}/`:

```
experiences/Customer_Portal/
├── config/                    # ← Auto-generated. Avoid modifying directly.
│   ├── Customer_Portal.json   #   Site-level config (theme, features)
│   └── mainAppPage.json       #   App page config
├── routes/                    # ← Page routes (URL paths)
│   ├── home.json
│   └── contactSupport.json
├── views/                     # ← Page layouts and component placement
│   ├── home.json
│   ├── contactSupport.json
│   └── login.json
├── themes/                    # ← Branding (colors, fonts, logo)
│   └── Theme1.json
└── Customer_Portal.site       # ← Site definition file
```

**Safe to modify**: `routes/`, `views/`, `themes/`, the `.site` file.
**Avoid modifying**: `config/` files — these are auto-generated by Experience Builder and overwritten on publish. Changes made directly here often cause deployment conflicts.

## Deployment Order and Commands

Network and CustomSite must deploy before ExperienceBundle. Deploy in two steps:

```bash
# Step 1: Deploy Network + CustomSite + Profiles + Sharing Rules
sf project deploy start \
  --metadata Network:Customer_Portal \
  --metadata CustomSite:Customer_Portal \
  --metadata Profile:Customer_Portal\ Profile

# Step 2: Deploy ExperienceBundle + Navigation
sf project deploy start \
  --metadata ExperienceBundle:Customer_Portal \
  --metadata NavigationMenu:Customer_Portal_Navigation
```

Or deploy the full source directory if your package structure keeps them together — Salesforce resolves internal ordering for source deploys:

```bash
# Full source deploy handles ordering automatically
sf project deploy start --source-dir force-app --test-level RunLocalTests
```

**After deploying**, publish the site to make changes live:

```bash
sf community publish --name "Customer Portal"
```

## Network Metadata Configuration

The Network metadata (`force-app/main/default/networks/{SiteName}.network`) controls critical site settings:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<Network xmlns="http://soap.sforce.com/2006/04/metadata">
    <allowMembersToFlag>true</allowMembersToFlag>
    <branding>
        <loginFooterText>Welcome to Customer Portal</loginFooterText>
    </branding>
    <changePasswordTemplate>unfiled$public/CommunityChangePasswordEmailTemplate</changePasswordTemplate>
    <communityRoles/>
    <emailSenderAddress>noreply@example.com</emailSenderAddress>
    <emailSenderName>Customer Portal</emailSenderName>
    <enableGuestChatter>false</enableGuestChatter>
    <enableInvitation>false</enableInvitation>
    <forgotPasswordTemplate>unfiled$public/CommunityForgotPasswordEmailTemplate</forgotPasswordTemplate>
    <guestMemberVisibility>false</guestMemberVisibility>
    <networkMemberGroups>
        <permissionSet>Customer_Portal_User</permissionSet>
        <profile>Customer Community Login User</profile>
    </networkMemberGroups>
    <networkPageOverrides>
        <changePasswordPageOverrideSetting>Designer</changePasswordPageOverrideSetting>
        <forgotPasswordPageOverrideSetting>Designer</forgotPasswordPageOverrideSetting>
        <homePageOverrideSetting>Designer</homePageOverrideSetting>
        <loginPageOverrideSetting>Designer</loginPageOverrideSetting>
        <selfRegProfilePageOverrideSetting>Designer</selfRegProfilePageOverrideSetting>
    </networkPageOverrides>
    <selfRegistration>false</selfRegistration>
    <status>Live</status>
    <tabs>
        <defaultTab>home</defaultTab>
    </tabs>
    <urlPathPrefix>customers</urlPathPrefix>
    <welcomeTemplate>unfiled$public/CommunityWelcomeEmailTemplate</welcomeTemplate>
</Network>
```

### Key Network Fields

| Field | Purpose | Pitfall |
|-------|---------|--------|
| `status` | `Live` or `DownForMaintenance` | Must be `Live` for site to be accessible. Deploy as `DownForMaintenance` first in fresh orgs, then switch to `Live` after ExperienceBundle deploys. |
| `networkMemberGroups` | Which profiles/permission sets can access the site | Missing profiles here = users can't log in |
| `urlPathPrefix` | The `/prefix/` in the site URL | Changing this breaks bookmarks and integrations |
| `selfRegistration` | Enable/disable self-reg | Requires `selfRegProfile` to be set |
| `emailSenderAddress` | From address for community emails | Must be an org-verified email address |

## Guest User Profile Configuration

Every Experience Cloud site gets an auto-created guest user profile named `{SiteName} Profile`. This profile controls what unauthenticated visitors can see and do.

### Naming Convention

The guest user profile name follows the pattern: **`{Site Label} Profile`** (with spaces, matching the site's label, not its API name).

For a site labeled "Customer Portal", the guest user profile is `Customer Portal Profile`.

### Setting Guest User Permissions

Grant permissions via the Profile metadata. Guest users need explicit object and field permissions for anything they should access:

```xml
<!-- In Profile: Customer_Portal_Profile -->
<objectPermissions>
    <allowCreate>false</allowCreate>
    <allowDelete>false</allowDelete>
    <allowEdit>false</allowEdit>
    <allowRead>true</allowRead>
    <modifyAllRecords>false</modifyAllRecords>
    <object>Knowledge__kav</object>
    <viewAllRecords>false</viewAllRecords>
</objectPermissions>
<fieldPermissions>
    <editable>false</editable>
    <field>Knowledge__kav.Title</field>
    <readable>true</readable>
</fieldPermissions>
<fieldPermissions>
    <editable>false</editable>
    <field>Knowledge__kav.Summary</field>
    <readable>true</readable>
</fieldPermissions>
```

### Guest User Apex Access

Guest users need explicit Apex class access for any `@AuraEnabled` controllers they invoke:

```xml
<!-- In Profile: Customer_Portal_Profile -->
<classAccesses>
    <apexClass>KnowledgeArticleController</apexClass>
    <enabled>true</enabled>
</classAccesses>
```

### Guest User Security Considerations

- Guest users run `without sharing` by default — they have no User record for sharing evaluation
- **Always validate data access in Apex** for guest-facing controllers. Don't rely on profile-level security alone.
- Restrict guest profile to read-only on non-essential objects
- Never grant guest users Modify All or View All on any object
- Set `guestMemberVisibility` to `false` in Network metadata unless guest users need to see other community members

## Sharing Rules for Guest Access

Guest users belong to a public group named `{SiteName} Site Guest User`. Create criteria-based sharing rules to expose specific records:

```xml
<!-- SharingRules for Knowledge__kav -->
<sharingCriteriaRules>
    <fullName>Knowledge__kav.Guest_Published_Articles</fullName>
    <accessLevel>Read</accessLevel>
    <booleanFilter/>
    <label>Guest Published Articles</label>
    <sharedTo>
        <group>Customer_Portal_Site_Guest_User</group>
    </sharedTo>
    <criteriaItems>
        <field>PublishStatus</field>
        <operation>equals</operation>
        <value>Online</value>
    </criteriaItems>
</sharingCriteriaRules>
```

**Public group naming convention**: `{SiteApiName}_Site_Guest_User` (underscores, no spaces).

## Permission Sets for Community Users

Community user permission sets differ from internal ones:

- Assign via `networkMemberGroups` in Network metadata (not via `sf org assign permset`)
- Community users authenticate through the site — they don't have the Salesforce app
- Use the `Customer Community Login User` or `Customer Community User` license profiles
- Create permission sets specific to the community user persona — don't share internal admin perm sets

## NavigationMenu and NavigationMenuItem

```xml
<!-- NavigationMenu -->
<NavigationMenu xmlns="http://soap.sforce.com/2006/04/metadata">
    <container>Customer_Portal</container>
    <containerType>Network</containerType>
    <label>Default Navigation</label>
    <navigationMenuItem>
        <label>Home</label>
        <position>0</position>
        <publiclyAvailable>true</publiclyAvailable>
        <target>home</target>
        <type>InternalLink</type>
    </navigationMenuItem>
    <navigationMenuItem>
        <label>Contact Us</label>
        <position>1</position>
        <publiclyAvailable>true</publiclyAvailable>
        <target>contact-support</target>
        <type>InternalLink</type>
    </navigationMenuItem>
</NavigationMenu>
```

**Gotcha**: NavigationMenuItems reference page routes. If the referenced route doesn't exist in the ExperienceBundle, deployment fails silently or the menu item appears broken. Always deploy ExperienceBundle pages before or alongside NavigationMenu.

## .forceignore Recommendations

ExperienceBundles generate constant diffs from auto-generated files. Add these to `.forceignore` to reduce noise:

```
# Experience Cloud auto-generated files
**/experiences/**/config/**
**/experiences/**/routes/loginPage.json
**/experiences/**/views/login.json

# Keep these tracked (remove from .forceignore if you need them):
# **/experiences/**/routes/**
# **/experiences/**/views/**
# **/experiences/**/themes/**
```

**Trade-off**: Ignoring `config/` means you manage site configuration in Experience Builder (the UI) rather than in source. This avoids deployment conflicts but means config doesn't travel with your source. Most teams accept this trade-off. If you need config in source (e.g., for CI/CD), remove the `config/` line and be prepared to resolve merge conflicts.

## LWR Sites (Build Your Own)

LWR sites use a different metadata format than Aura-based Experience Cloud sites. The `DigitalExperienceBundle` metadata type replaces `ExperienceBundle`, and the directory structure uses `digitalExperiences/` instead of `experiences/`.

### Creating an LWR Site via CLI

```bash
# Digital Experiences must be enabled via Setup UI first — CLI can't enable it
sf community create --name "Site Name" --template-name "Build Your Own (LWR)" --url-path-prefix myprefix

# Retrieve site metadata (use DigitalExperienceBundle, not ExperienceBundle)
sf project retrieve start --metadata "DigitalExperienceBundle"

# Publish after deploy
sf community publish --name "Site Name"
```

**Gotcha**: The URL path prefix you request may not match what Salesforce assigns (e.g., requested `dte`, got `dtevforcesite`). Always retrieve after creation to confirm the actual prefix.

### LWR Directory Structure

After retrieval, LWR sites land in `force-app/main/default/digitalExperiences/site/{SiteName}/`:

```
digitalExperiences/site/MySite/
├── MySite.digitalExperience-meta.xml
├── sfdc_cms__route/          # URL routing (pageAccess lives here)
│   ├── home/content.json
│   └── contact_us/content.json
├── sfdc_cms__view/           # Page content & component placement
│   ├── home/content.json
│   └── contact_us/content.json
├── sfdc_cms__site/           # Site-level config
├── sfdc_cms__theme/          # Theme settings
├── sfdc_cms__themeLayout/    # Header/footer layouts
├── sfdc_cms__brandingSet/    # Branding/colors
└── sfdc_cms__styles/         # CSS
```

### Route-Level Guest Access (LWR)

The site-level `authenticationType` is locked to `"AUTHENTICATED"` and **cannot be changed via metadata deploy**. Salesforce rejects the deploy with: `"You can't switch to unauthenticated sites."`

Instead, set `"pageAccess": "Public"` on individual route metadata files:

```json
{
  "type": "sfdc_cms__route",
  "title": "Home",
  "contentBody": {
    "activeViewId": "home",
    "configurationTags": [],
    "pageAccess": "Public",
    "routeType": "home",
    "urlPrefix": ""
  },
  "urlName": "home"
}
```

The default value `"UseParent"` inherits the site-level auth wall, forcing login. Set to `"Public"` for each page that guest users should access.

### Adding an LWC to an LWR Page

Place your component inside a section > column structure in the view's `content.json`:

```json
{
  "attributes": {},
  "definition": "c:myComponentName",
  "id": "<uuid>",
  "type": "component"
}
```

Components need `lightningCommunity__Page` and `lightningCommunity__Default` in their `.js-meta.xml` targets to appear in Experience Builder for LWR sites.

### Guest User Permission Sets (LWR)

Guest user licenses are restrictive — only **Create + Read** are allowed:

- No Edit, Delete, View All, or Modify All on objects
- Field-level permissions must be `editable: false` (read-only)
- Apex controllers running `without sharing` handle all DML in system context

Create a separate `*_Guest_Access` permission set with minimal Create + Read permissions. Don't assign the admin-level perm set to the guest user.

### Granting Guest User Apex Class Access

For scratch orgs and POCs, grant Apex class access to the guest user profile via anonymous Apex using `SetupEntityAccess`:

```apex
Id profilePermSetId = [SELECT Id FROM PermissionSet
    WHERE IsOwnedByProfile = true
    AND ProfileId IN (SELECT Id FROM Profile WHERE Name = '<SiteName> Profile')
    LIMIT 1].Id;

List<ApexClass> classes = [SELECT Id, Name FROM ApexClass WHERE Name IN :classNames];
for (ApexClass cls : classes) {
    insert new SetupEntityAccess(SetupEntityId = cls.Id, ParentId = profilePermSetId);
}
```

For persistent projects, use profile metadata `classAccesses` instead (see Guest User Apex Access section above).

### LWR-Specific Gotchas

- `geoBotsAllowed` field in site `content.json` is rejected on deploy — remove it if present
- Digital Experiences must be enabled via Setup UI before `sf community create` works
- LWR sites use `DigitalExperienceBundle` metadata type, not `ExperienceBundle`
- The `sfdc_cms__*` folder naming is fixed — don't rename or restructure

## Common Deployment Errors

| Error | Cause | Fix |
|-------|-------|-----|
| `Cannot deploy ExperienceBundle — Network not found` | Deploying ExperienceBundle before the Network exists | Deploy Network first, or use full source deploy |
| `Guest user profile not found` | Profile name doesn't match site label | Check site label (not API name) and use `{Label} Profile` |
| `NavigationMenuItem target not found` | Menu item references a route that doesn't exist | Deploy ExperienceBundle first, then NavigationMenu |
| `Network emailSenderAddress is not valid` | Email address not verified in org | Add and verify the email address in Setup → Organization-Wide Email Addresses |
| `Unable to deploy ExperienceBundle — site not published` | First-time deploy to a fresh org | Create the site first (via UI or `sf community create`), then deploy the bundle |
| `ExperienceBundle conflicts detected` | Auto-generated files changed in both source and org | Reset to org state: retrieve, accept org version, re-apply your changes |
| `DuplicateValue` on NavigationMenu | Menu with same name already exists | Delete existing menu in org first, or retrieve and modify instead of creating new |
